# 요구사항 문서: 고객 상세 페이지 재구성 및 캘린더 연동

## 소개

현재 고객 상세 페이지(`/customers/[id]`)는 기본 정보(이름, 이메일, 전화번호, 주소, 메모, 등급)와 메타 정보(등록일/수정일)만 표시하고 있다. 이 기능은 고객 상세 페이지를 탭 기반 UI로 재구성하여, 캘린더 이벤트·미팅 기록·시험 접수·세금계산서 일정·진행 단계·의뢰자 등 모든 관련 데이터를 한 화면에서 조회·관리할 수 있도록 한다. 특히 캘린더에 등록된 이벤트가 해당 고객사 상세 페이지에 자동으로 연동되어 표시되어야 한다.

## 용어 정의

- **고객_상세_페이지**: `/customers/[id]` 경로에서 특정 고객사의 모든 관련 정보를 표시하는 프론트엔드 페이지
- **탭_네비게이션**: 고객 상세 페이지 내에서 각 데이터 영역(개요, 캘린더, 미팅, 시험접수, 세금계산서, 의뢰자)을 전환하는 UI 컴포넌트
- **개요_탭**: 고객 기본 정보와 진행 단계, 최근 활동 요약을 표시하는 기본 탭
- **캘린더_탭**: 해당 고객사에 연결된 캘린더 이벤트를 월간/주간/일간 뷰로 표시하는 탭
- **미팅_기록_탭**: 해당 고객사의 미팅·통화·이메일·방문 기록을 목록으로 표시하는 탭
- **시험_접수_탭**: 해당 고객사의 시험 접수 현황을 표시하는 탭
- **세금계산서_탭**: 해당 고객사의 세금계산서 발행 일정을 표시하는 탭
- **의뢰자_탭**: 해당 고객사에 소속된 의뢰자(담당자) 목록을 표시하는 탭
- **CalendarView_컴포넌트**: 기존에 구현된 캘린더 뷰 컴포넌트로, `customerId` prop을 통해 고객사별 이벤트 필터링을 지원
- **고객_데이터_API**: `customer-data-api.ts`에 정의된 고객 하위 데이터 조회 API 모음

## 요구사항

### 요구사항 1: 탭 기반 페이지 레이아웃

**사용자 스토리:** 영업 담당자로서, 고객 상세 페이지에서 탭을 통해 관련 데이터 영역을 전환하고 싶다. 그래야 한 화면에서 고객의 모든 정보를 효율적으로 탐색할 수 있다.

#### 인수 조건

1. THE 고객_상세_페이지 SHALL 상단에 고객 기본 정보(회사명, 등급, 담당자, 연락처)를 요약 헤더로 표시한다
2. THE 탭_네비게이션 SHALL 개요, 캘린더, 미팅 기록, 시험 접수, 세금계산서, 의뢰자 총 6개의 탭을 제공한다
3. WHEN 사용자가 탭을 클릭하면, THE 고객_상세_페이지 SHALL 해당 탭의 콘텐츠 영역으로 전환한다
4. THE 고객_상세_페이지 SHALL 페이지 최초 로드 시 개요_탭을 기본으로 표시한다
5. THE 탭_네비게이션 SHALL 모바일 화면(768px 미만)에서 가로 스크롤 가능한 형태로 표시한다

### 요구사항 2: 개요 탭

**사용자 스토리:** 영업 담당자로서, 고객 상세 페이지에 진입했을 때 고객의 전체적인 현황을 한눈에 파악하고 싶다. 그래야 빠르게 상황을 이해하고 다음 행동을 결정할 수 있다.

#### 인수 조건

1. THE 개요_탭 SHALL 고객 기본 정보(회사명, 사업자번호, 주소, 담당자, 이메일, 전화번호, 메모)를 표시한다
2. THE 개요_탭 SHALL 고객 등급을 표시하고 등급 변경 기능을 제공한다
3. THE 개요_탭 SHALL 현재 진행 단계(ProgressStage) 정보를 요약하여 표시한다
4. THE 개요_탭 SHALL 최근 미팅 기록 3건을 요약 목록으로 표시한다
5. THE 개요_탭 SHALL 다가오는 캘린더 이벤트 3건을 요약 목록으로 표시한다
6. WHEN 사용자가 요약 목록의 "더보기" 링크를 클릭하면, THE 고객_상세_페이지 SHALL 해당 데이터의 전체 탭으로 전환한다

### 요구사항 3: 캘린더 연동 탭

**사용자 스토리:** 영업 담당자로서, 고객 상세 페이지에서 해당 고객사와 관련된 캘린더 이벤트를 조회하고 새 이벤트를 등록하고 싶다. 그래야 캘린더와 고객 관리가 자연스럽게 연동된다.

#### 인수 조건

1. THE 캘린더_탭 SHALL 기존 CalendarView_컴포넌트를 해당 고객사의 customerId로 필터링하여 표시한다
2. WHEN 사용자가 캘린더_탭에서 새 이벤트를 생성하면, THE 캘린더_탭 SHALL 해당 고객사의 customer_id를 자동으로 설정한다
3. WHEN 캘린더 페이지에서 고객사를 선택하여 이벤트를 등록하면, THE 캘린더_탭 SHALL 해당 이벤트를 고객 상세 페이지의 캘린더_탭에서 조회할 수 있도록 한다
4. THE 캘린더_탭 SHALL 이벤트 유형(미팅, 세금계산서, 마감일, 리마인더, 기타)별로 색상을 구분하여 표시한다
5. WHEN 사용자가 캘린더 이벤트를 클릭하면, THE 캘린더_탭 SHALL 이벤트 상세 정보를 표시한다

### 요구사항 4: 미팅 기록 탭

**사용자 스토리:** 영업 담당자로서, 고객과의 모든 미팅·통화·이메일·방문 기록을 시간순으로 확인하고 싶다. 그래야 고객과의 소통 이력을 추적할 수 있다.

#### 인수 조건

1. THE 미팅_기록_탭 SHALL meetingRecordApi.getByCustomerId를 호출하여 해당 고객사의 미팅 기록 목록을 표시한다
2. THE 미팅_기록_탭 SHALL 각 기록의 유형(미팅, 통화, 이메일, 방문), 날짜, 제목, 참석자를 목록에 표시한다
3. THE 미팅_기록_탭 SHALL 기록을 날짜 기준 최신순으로 정렬하여 표시한다
4. WHEN 사용자가 미팅 기록 항목을 클릭하면, THE 미팅_기록_탭 SHALL 해당 기록의 상세 내용(내용, 후속 조치, 첨부파일)을 표시한다
5. IF 해당 고객사의 미팅 기록이 없으면, THEN THE 미팅_기록_탭 SHALL "등록된 미팅 기록이 없습니다" 빈 상태 메시지를 표시한다

### 요구사항 5: 시험 접수 탭

**사용자 스토리:** 영업 담당자로서, 고객사의 시험 접수 현황과 진행 상태를 확인하고 싶다. 그래야 시험 진행 상황을 고객에게 안내할 수 있다.

#### 인수 조건

1. THE 시험_접수_탭 SHALL testReceptionApi.getByCustomerId를 호출하여 해당 고객사의 시험 접수 목록을 표시한다
2. THE 시험_접수_탭 SHALL 각 접수의 시험번호, 시험제목, 상태(접수/진행중/완료/취소), 접수일, 예상완료일을 표시한다
3. THE 시험_접수_탭 SHALL 상태별 배지 색상을 구분하여 표시한다
4. THE 시험_접수_탭 SHALL 금액 정보(총액, 납부액, 잔액)를 표시한다
5. IF 해당 고객사의 시험 접수가 없으면, THEN THE 시험_접수_탭 SHALL "등록된 시험 접수가 없습니다" 빈 상태 메시지를 표시한다

### 요구사항 6: 세금계산서 일정 탭

**사용자 스토리:** 영업 담당자로서, 고객사의 세금계산서 발행 일정과 납부 상태를 확인하고 싶다. 그래야 미수금 관리와 발행 일정을 추적할 수 있다.

#### 인수 조건

1. THE 세금계산서_탭 SHALL invoiceScheduleApi.getByCustomerId를 호출하여 해당 고객사의 세금계산서 일정 목록을 표시한다
2. THE 세금계산서_탭 SHALL 각 일정의 금액, 발행예정일, 실제발행일, 상태(대기/발행/납부/연체)를 표시한다
3. THE 세금계산서_탭 SHALL 상태별 배지 색상을 구분하여 표시한다(연체는 빨간색)
4. THE 세금계산서_탭 SHALL 일정을 발행예정일 기준으로 정렬하여 표시한다
5. IF 해당 고객사의 세금계산서 일정이 없으면, THEN THE 세금계산서_탭 SHALL "등록된 세금계산서 일정이 없습니다" 빈 상태 메시지를 표시한다

### 요구사항 7: 의뢰자 관리 탭

**사용자 스토리:** 영업 담당자로서, 고객사에 소속된 의뢰자(담당자) 목록을 확인하고 싶다. 그래야 적절한 담당자에게 연락할 수 있다.

#### 인수 조건

1. THE 의뢰자_탭 SHALL requesterApi.getByCustomerId를 호출하여 해당 고객사의 의뢰자 목록을 표시한다
2. THE 의뢰자_탭 SHALL 각 의뢰자의 이름, 직책, 부서, 전화번호, 이메일을 표시한다
3. THE 의뢰자_탭 SHALL 주 담당자(is_primary)를 시각적으로 구분하여 표시한다
4. THE 의뢰자_탭 SHALL 비활성 의뢰자(is_active=false)를 흐리게 표시하여 구분한다
5. IF 해당 고객사의 의뢰자가 없으면, THEN THE 의뢰자_탭 SHALL "등록된 의뢰자가 없습니다" 빈 상태 메시지를 표시한다

### 요구사항 8: 데이터 로딩 및 오류 처리

**사용자 스토리:** 영업 담당자로서, 각 탭의 데이터가 로딩 중일 때 로딩 상태를 확인하고, 오류 발생 시 적절한 안내를 받고 싶다. 그래야 시스템 상태를 이해하고 적절히 대응할 수 있다.

#### 인수 조건

1. WHILE 탭 데이터가 로딩 중이면, THE 고객_상세_페이지 SHALL 스켈레톤 UI를 표시한다
2. IF API 호출이 실패하면, THEN THE 고객_상세_페이지 SHALL 오류 메시지와 재시도 버튼을 표시한다
3. WHEN 사용자가 탭을 전환하면, THE 고객_상세_페이지 SHALL 해당 탭의 데이터를 지연 로딩(lazy loading)한다
4. THE 고객_상세_페이지 SHALL 이미 로딩된 탭 데이터를 캐싱하여 탭 재전환 시 재요청하지 않는다
