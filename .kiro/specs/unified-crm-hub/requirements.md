# 요구사항 문서: 통합 CRM 허브 - 고객사 중심 데이터 연동

## 소개

현재 시스템은 리드 관리(`/leads`)와 고객사 관리(`/customers`)가 별도 모듈로 분리되어 있으며, 고객 상세 페이지에서 견적서·계약·리드 활동 등 핵심 CRM 데이터를 조회할 수 없다. 또한 캘린더·견적서·계약 등 다른 모듈에서 생성된 데이터가 고객사 상세 페이지에 자동으로 연동되지 않는다.

본 기능은 리드 관리를 고객사 관리 모듈로 통합하고, 고객사 상세 페이지에서 해당 업체의 모든 CRM 데이터(견적서, 계약, 리드 활동, 상담기록 등)를 조회할 수 있도록 하며, 다른 모듈에서 생성된 데이터가 양방향으로 연동되는 통합 CRM 허브를 구축한다.

## 용어 정의

- **통합_고객사_모듈**: `/customers` 경로에서 리드와 고객을 등급(LEAD, PROSPECT, CUSTOMER, VIP, INACTIVE)으로 구분하여 통합 관리하는 단일 모듈
- **고객사_상세_페이지**: `/customers/[id]` 경로에서 특정 고객사의 모든 CRM 연관 데이터를 탭 기반으로 표시하는 페이지
- **사이드바**: 좌측 네비게이션 메뉴 (`Sidebar.tsx`)
- **견적서_탭**: 고객사 상세 페이지에서 해당 업체에 연결된 견적서(독성·효력·임상병리) 목록을 표시하는 탭
- **계약_탭**: 고객사 상세 페이지에서 해당 업체에 연결된 계약 목록을 표시하는 탭
- **리드_활동_탭**: 고객사 상세 페이지에서 해당 업체의 리드 활동 이력(상태 변경, 메모 등)을 표시하는 탭
- **상담기록_탭**: 고객사 상세 페이지에서 해당 업체에 연결된 상담기록 목록을 표시하는 탭
- **통합_활동_타임라인**: 고객사의 모든 CRM 활동(견적서 생성, 계약 체결, 미팅, 캘린더 이벤트, 리드 상태 변경 등)을 시간순으로 통합 표시하는 뷰
- **양방향_연동**: 다른 모듈(캘린더, 견적서, 계약 등)에서 고객사를 선택하여 데이터를 생성하면, 해당 고객사 상세 페이지에서도 해당 데이터가 자동으로 조회되는 구조
- **고객사_데이터_API**: 고객사 하위 데이터를 조회하는 백엔드 API 엔드포인트 모음 (`/api/customer-data/customers/:id/*`)
- **Quotation_API**: 견적서 조회 API (`/api/quotations?customerId=xxx`)
- **Contract_API**: 계약 조회 API (`/api/contracts?customerId=xxx`)
- **Consultation_API**: 상담기록 조회 API (`/api/consultations?customerId=xxx`)
- **Lead_Activity_API**: 리드 활동 조회 API (`/api/leads/:id/activities`)

## 요구사항

### 요구사항 1: 사이드바 메뉴 통합

**사용자 스토리:** 영업 담당자로서, 사이드바에서 리드 관리와 고객사 관리가 하나의 메뉴로 통합되어 있길 원한다. 그래야 별도 모듈을 오가지 않고 한 곳에서 모든 고객사를 관리할 수 있다.

#### 인수 조건

1. THE 사이드바 SHALL 영업관리 그룹에서 "리드 관리" 메뉴 항목을 제거한다
2. THE 사이드바 SHALL 영업관리 그룹에서 "고객사 관리" 메뉴 항목을 유지하며, 해당 메뉴가 통합 고객사 목록(`/customers`)으로 연결되도록 한다
3. THE 사이드바 SHALL 영업관리 그룹에서 "파이프라인" 메뉴 항목(`/pipeline`)을 기존과 동일하게 유지한다
4. WHEN 사용자가 기존 `/leads` URL로 직접 접근하면, THE 통합_고객사_모듈 SHALL `/customers?type=lead` 페이지로 리다이렉트한다

### 요구사항 2: 통합 고객사 목록에서 리드 클릭 시 고객사 상세로 이동

**사용자 스토리:** 영업 담당자로서, 통합 고객사 목록에서 리드 항목을 클릭했을 때 별도의 리드 상세 페이지(`/leads/{id}`)가 아닌 고객사 상세 페이지(`/customers/{id}`)로 이동하길 원한다. 그래야 모든 고객사 정보를 하나의 상세 페이지에서 관리할 수 있다.

#### 인수 조건

1. WHEN 사용자가 통합 고객사 목록에서 리드 유형 항목을 클릭하면, THE 통합_고객사_모듈 SHALL 해당 리드에 연결된 고객사의 상세 페이지(`/customers/{customerId}`)로 이동한다
2. IF 리드에 연결된 고객사(customerId)가 없으면, THEN THE 통합_고객사_모듈 SHALL 해당 리드의 정보를 기반으로 고객사를 자동 생성한 후 고객사 상세 페이지로 이동한다
3. THE 통합_고객사_모듈 SHALL 리드 항목 카드에 리드 고유 정보(리드번호, 소스, 예상금액, 파이프라인 단계)를 표시한다

### 요구사항 3: 고객사 상세 페이지 - 견적서 탭 추가

**사용자 스토리:** 영업 담당자로서, 고객사 상세 페이지에서 해당 업체에 연결된 모든 견적서를 조회하고 싶다. 그래야 견적 목록 페이지를 별도로 방문하지 않고도 해당 업체의 견적 현황을 파악할 수 있다.

#### 인수 조건

1. THE 고객사_상세_페이지 SHALL 기존 탭(개요, 캘린더, 미팅 기록, 시험 접수, 세금계산서, 의뢰자) 외에 "견적서" 탭을 추가한다
2. WHEN 사용자가 견적서_탭을 선택하면, THE 견적서_탭 SHALL Quotation_API를 호출하여 해당 고객사의 customerId로 필터링된 견적서 목록을 표시한다
3. THE 견적서_탭 SHALL 각 견적서의 견적번호, 견적유형(독성/효력/임상병리), 프로젝트명, 상태(초안/발송/승인/거절), 총액, 생성일을 표시한다
4. WHEN 사용자가 견적서 항목을 클릭하면, THE 견적서_탭 SHALL 해당 견적서의 상세 페이지로 이동한다
5. IF 해당 고객사의 견적서가 없으면, THEN THE 견적서_탭 SHALL "등록된 견적서가 없습니다" 빈 상태 메시지를 표시한다

### 요구사항 4: 고객사 상세 페이지 - 계약 탭 추가

**사용자 스토리:** 영업 담당자로서, 고객사 상세 페이지에서 해당 업체에 연결된 모든 계약을 조회하고 싶다. 그래야 계약 관리 페이지를 별도로 방문하지 않고도 해당 업체의 계약 현황을 파악할 수 있다.

#### 인수 조건

1. THE 고객사_상세_페이지 SHALL 기존 탭 외에 "계약" 탭을 추가한다
2. WHEN 사용자가 계약_탭을 선택하면, THE 계약_탭 SHALL Contract_API를 호출하여 해당 고객사의 customerId로 필터링된 계약 목록을 표시한다
3. THE 계약_탭 SHALL 각 계약의 계약번호, 계약유형, 제목, 상태(초안/활성/완료/취소), 계약금액, 시작일, 종료일을 표시한다
4. WHEN 사용자가 계약 항목을 클릭하면, THE 계약_탭 SHALL 해당 계약의 상세 페이지(`/contracts/{contractId}`)로 이동한다
5. IF 해당 고객사의 계약이 없으면, THEN THE 계약_탭 SHALL "등록된 계약이 없습니다" 빈 상태 메시지를 표시한다

### 요구사항 5: 고객사 상세 페이지 - 리드 활동 탭 추가

**사용자 스토리:** 영업 담당자로서, 고객사 상세 페이지에서 해당 업체가 리드 단계였을 때의 활동 이력(상태 변경, 메모, 소스 등)을 조회하고 싶다. 그래야 리드에서 고객으로 전환된 과정을 추적할 수 있다.

#### 인수 조건

1. THE 고객사_상세_페이지 SHALL 해당 고객사에 연결된 리드(linkedLead)가 존재하는 경우 "리드 활동" 탭을 표시한다
2. WHEN 사용자가 리드_활동_탭을 선택하면, THE 리드_활동_탭 SHALL Lead_Activity_API를 호출하여 해당 리드의 활동 이력을 표시한다
3. THE 리드_활동_탭 SHALL 리드 기본 정보(리드번호, 소스, 현재 상태, 파이프라인 단계, 예상금액, 전환일)를 요약 카드로 표시한다
4. THE 리드_활동_탭 SHALL 리드 활동 이력을 시간순으로 정렬하여 표시한다
5. IF 해당 고객사에 연결된 리드가 없으면, THEN THE 고객사_상세_페이지 SHALL 리드_활동_탭을 표시하지 않는다

### 요구사항 6: 고객사 상세 페이지 - 상담기록 탭 추가

**사용자 스토리:** 영업 담당자로서, 고객사 상세 페이지에서 해당 업체에 연결된 상담기록을 조회하고 싶다. 그래야 상담기록 페이지를 별도로 방문하지 않고도 해당 업체의 상담 이력을 파악할 수 있다.

#### 인수 조건

1. THE 고객사_상세_페이지 SHALL 기존 탭 외에 "상담기록" 탭을 추가한다
2. WHEN 사용자가 상담기록_탭을 선택하면, THE 상담기록_탭 SHALL Consultation_API를 호출하여 해당 고객사의 customerId로 필터링된 상담기록 목록을 표시한다
3. THE 상담기록_탭 SHALL 각 상담기록의 상담일, 상담유형, 제목, 상담내용 요약, 후속조치를 표시한다
4. IF 해당 고객사의 상담기록이 없으면, THEN THE 상담기록_탭 SHALL "등록된 상담기록이 없습니다" 빈 상태 메시지를 표시한다


### 요구사항 7: 고객사 상세 페이지 - 개요 탭 통합 활동 타임라인

**사용자 스토리:** 영업 담당자로서, 고객사 상세 페이지의 개요 탭에서 해당 업체의 모든 CRM 활동을 시간순으로 통합하여 확인하고 싶다. 그래야 한눈에 고객사와의 전체 상호작용 이력을 파악할 수 있다.

#### 인수 조건

1. THE 개요_탭 SHALL 기존 정보(기본 정보, 진행 단계, 최근 미팅, 다가오는 이벤트) 외에 "최근 활동" 섹션을 추가한다
2. THE 통합_활동_타임라인 SHALL 견적서 생성/상태변경, 계약 체결/상태변경, 미팅 기록, 캘린더 이벤트, 리드 상태 변경, 상담기록을 시간순으로 통합하여 최근 10건을 표시한다
3. THE 통합_활동_타임라인 SHALL 각 활동 항목에 활동 유형 아이콘, 활동 내용 요약, 발생 일시를 표시한다
4. WHEN 사용자가 타임라인 항목을 클릭하면, THE 통합_활동_타임라인 SHALL 해당 활동의 상세 정보가 있는 탭으로 전환한다
5. THE 개요_탭 SHALL 견적서 요약(총 건수, 최근 견적서)과 계약 요약(총 건수, 활성 계약)을 요약 카드로 표시한다

### 요구사항 8: 양방향 데이터 연동 - 견적서 모듈

**사용자 스토리:** 영업 담당자로서, 견적서 작성 시 고객사를 선택하면 해당 견적서가 고객사 상세 페이지의 견적서 탭에서 자동으로 조회되길 원한다. 그래야 데이터를 수동으로 연결하지 않아도 된다.

#### 인수 조건

1. WHEN 견적서 작성 페이지(`/quotations/new`)에서 고객사를 선택하여 견적서를 저장하면, THE Quotation_API SHALL 해당 견적서의 customerId 필드에 선택된 고객사 ID를 저장한다
2. WHEN 고객사 상세 페이지의 견적서_탭이 로드되면, THE 견적서_탭 SHALL Quotation_API에 customerId 파라미터를 전달하여 해당 고객사의 견적서만 조회한다
3. THE 견적서_탭 SHALL 독성시험 견적서, 효력시험 견적서, 임상병리 견적서를 모두 통합하여 표시한다

### 요구사항 9: 양방향 데이터 연동 - 캘린더 모듈

**사용자 스토리:** 영업 담당자로서, 캘린더에서 특정 업체에 대한 미팅 일정을 등록하면 해당 업체의 고객사 상세 페이지 캘린더 탭에서 자동으로 조회되길 원한다. 그래야 캘린더와 고객사 관리가 자연스럽게 연동된다.

#### 인수 조건

1. WHEN 캘린더 페이지(`/calendar`)에서 고객사를 선택하여 이벤트를 생성하면, THE 고객사_데이터_API SHALL 해당 이벤트의 customer_id 필드에 선택된 고객사 ID를 저장한다
2. WHEN 고객사 상세 페이지의 캘린더 탭이 로드되면, THE 캘린더 탭 SHALL 해당 고객사의 customer_id로 필터링된 이벤트만 표시한다
3. WHEN 고객사 상세 페이지의 개요 탭이 로드되면, THE 개요_탭 SHALL 해당 고객사의 다가오는 캘린더 이벤트 3건을 요약으로 표시한다

### 요구사항 10: 양방향 데이터 연동 - 계약 모듈

**사용자 스토리:** 영업 담당자로서, 계약 관리에서 특정 업체에 대한 계약을 생성하면 해당 업체의 고객사 상세 페이지 계약 탭에서 자동으로 조회되길 원한다. 그래야 계약 관리와 고객사 관리가 자연스럽게 연동된다.

#### 인수 조건

1. WHEN 계약 관리 페이지(`/contracts`)에서 고객사를 선택하여 계약을 생성하면, THE Contract_API SHALL 해당 계약의 customerId 필드에 선택된 고객사 ID를 저장한다
2. WHEN 고객사 상세 페이지의 계약_탭이 로드되면, THE 계약_탭 SHALL Contract_API에 customerId 파라미터를 전달하여 해당 고객사의 계약만 조회한다
3. THE 개요_탭 SHALL 해당 고객사의 활성 계약 건수와 최근 계약 정보를 요약 카드로 표시한다

### 요구사항 11: 통합 고객사 데이터 조회 API

**사용자 스토리:** 시스템 개발자로서, 고객사 상세 페이지에서 필요한 모든 연관 데이터(견적서, 계약, 리드 활동, 상담기록)를 효율적으로 조회하는 API가 필요하다. 그래야 프론트엔드에서 여러 API를 개별 호출하지 않고 통합된 데이터를 받을 수 있다.

#### 인수 조건

1. THE 고객사_데이터_API SHALL `GET /api/customer-data/customers/:id/quotations` 엔드포인트를 제공하여 해당 고객사의 견적서 목록을 반환한다
2. THE 고객사_데이터_API SHALL `GET /api/customer-data/customers/:id/contracts` 엔드포인트를 제공하여 해당 고객사의 계약 목록을 반환한다
3. THE 고객사_데이터_API SHALL `GET /api/customer-data/customers/:id/lead-activities` 엔드포인트를 제공하여 해당 고객사에 연결된 리드의 활동 이력을 반환한다
4. THE 고객사_데이터_API SHALL `GET /api/customer-data/customers/:id/consultations` 엔드포인트를 제공하여 해당 고객사의 상담기록 목록을 반환한다
5. THE 고객사_데이터_API SHALL `GET /api/customer-data/customers/:id/activity-timeline` 엔드포인트를 제공하여 모든 CRM 활동을 시간순으로 통합한 타임라인 데이터를 반환한다
6. WHEN API 응답이 반환되면, THE 고객사_데이터_API SHALL 각 항목에 활동 유형(quotation, contract, meeting, calendar_event, lead_activity, consultation), 발생 일시, 요약 내용을 포함한다

### 요구사항 12: 데이터 로딩 및 오류 처리

**사용자 스토리:** 영업 담당자로서, 새로 추가된 탭(견적서, 계약, 리드 활동, 상담기록)의 데이터가 로딩 중일 때 로딩 상태를 확인하고, 오류 발생 시 적절한 안내를 받고 싶다.

#### 인수 조건

1. WHILE 탭 데이터가 로딩 중이면, THE 고객사_상세_페이지 SHALL 스켈레톤 UI를 표시한다
2. IF API 호출이 실패하면, THEN THE 고객사_상세_페이지 SHALL 오류 메시지와 재시도 버튼을 표시한다
3. WHEN 사용자가 탭을 전환하면, THE 고객사_상세_페이지 SHALL 해당 탭의 데이터를 지연 로딩(lazy loading)한다
4. THE 고객사_상세_페이지 SHALL 이미 로딩된 탭 데이터를 캐싱하여 탭 재전환 시 재요청하지 않는다
